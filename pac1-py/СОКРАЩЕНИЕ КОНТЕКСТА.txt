  В t30 step 11 отправляется 122k chars — это:                                                                                                                                                                                                            - System prompt (~30k) — с workspace rules включая Telegram.txt (16k)                                                                                                                                                                                   - Conversation history (~90k) — 10 предыдущих шагов с полными tool results                                                                                                                                                                            
  Каждый LLM call платит за весь накопленный контекст. Step 11 платит за content всех предыдущих 10 шагов.

  Как делают грамотные инженеры

  1. Не грузить data-файлы в system prompt

  Проблема: Bootstrap загрузил Telegram.txt (1025 строк, ~16k chars) в workspace_rules. Это DATA, не INSTRUCTION. Оно повторяется в каждом из 11 вызовов = 176k chars впустую.

  Решение: Лимит на размер файла в bootstrap. Если файл > 2000 chars — не грузить в rules, а оставить пометку "файл существует, прочитай через read tool когда нужно". Агент прочитает один раз, результат будет только в conversation_history этого
  шага.

  Экономия для t30: ~160k chars (16k × 10 повторений).

  2. Sliding window на conversation_history

  Проблема: Step 10 тащит все 9 предыдущих exchanges. Но агент уже записал ключевые находки в scratchpad (entity_graph, completed_steps).

  Решение: Хранить полные данные только последних 4-6 шагов. Старые шаги сжимать до одной строки: "step_3: read contacts/cont_007.json → found Ilse Sanders, acct_007, email verified".

  Scratchpad уже содержит всё важное — entity_graph, completed_steps. LLM не потеряет контекст.

  Экономия для t23: ~20-30k chars на поздних шагах.

  3. Truncate tool results в старых шагах

  Проблема: Когда агент прочитал JSON файл на step 3, полный content (~2k chars) остаётся в history навсегда. На step 10 он уже не нужен — данные в scratchpad.

  Решение: В conversation_history для шагов старше 4 — обрезать tool result до 200 chars + "[full content was processed, see scratchpad for extracted data]".

  4. Статичные блоки вынести из system prompt

  Проблема: Strategic context (checklist, risks, scope) — ~3k chars, статичный, одинаковый на каждом шаге.

  Решение: Передать один раз как первое сообщение в conversation_history, не в system prompt. Или: передать только на первом шаге, дальше ссылка "see strategic analysis above".

  Что самое безопасное и эффективное

  │          Решение          │       Экономия       │        Риск сломать        │ Сложность │
  ├───────────────────────────┼──────────────────────┼────────────────────────────┼───────────┤
  │ Лимит размера в bootstrap │ Высокая (t30: -160k) │ Низкий                     │ Низкая    │
  ├───────────────────────────┼──────────────────────┼────────────────────────────┼───────────┤
  │ Sliding window history    │ Средняя (-20-30k)    │ Средний — LLM может забыть │ Средняя   │
  ├───────────────────────────┼──────────────────────┼────────────────────────────┼───────────┤
  │ Truncate old results      │ Средняя (-15-20k)    │ Низкий — scratchpad хранит │ Низкая    │
  ├───────────────────────────┼──────────────────────┼────────────────────────────┼───────────┤
  │ Статику из prompt         │ Малая (-3k×N)        │ Низкий                     │ Низкая    │
  └───────────────────────────┴──────────────────────┴────────────────────────────┴───────────┘

  Моя рекомендация

  Начать с #1 (лимит в bootstrap) — самый большой выигрыш, минимальный риск. Telegram.txt не должен быть в system prompt. Затем #3 (truncate old results) — безопасно потому что scratchpad хранит извлечённые данные.

  Sliding window (#2) — потенциально опасен. LLM может потерять контекст и перечитать файлы. Лучше делать только если первые два не достаточны.
