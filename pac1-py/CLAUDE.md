# PAC1 Multi-Agent Benchmark — Рабочая инструкция

## Цель

Последовательно решить задачи PAC1 бенчмарка (t01–t10) с максимальным score.
Агент использует `LLM_PROVIDER=claude` — Claude Code CLI выступает мозгом системы.

## Главный цикл работы

При получении команды на прогон задач выполняй строго по циклу:

### Для каждой задачи tXX (от t01 до t10):

1. **Запуск**: `python main.py tXX`
2. **Проверка score**:
   - Score = 1.00 → задача решена, записать в отчёт, перейти к следующей
   - Score < 1.00 → перейти к шагу 3
3. **Анализ ошибки**: Прочитать score_detail из вывода. Определить корневую причину:
   - Агент неправильно назвал файл?
   - Агент пропустил шаг (distill, thread update, delete)?
   - Агент не понял инструкцию?
   - Ошибка парсинга / валидации?
4. **Исправление системы**: Внести универсальную правку в код агента (промпты, типы, оркестратор). Корректировать свой логический аппарат и подход, соблюдая все принципы из `benchmark_system_questions_answers.md`. **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО** добавлять частные значения — конкретные имена файлов, пути задачи, ID тасков, конкретные значения полей. Правка должна работать для ЛЮБОЙ задачи бенчмарка.
5. **Повторный запуск**: Вернуться к шагу 1 для этой же задачи.
6. **Повтор**: Цикл 1–5 до score = 1.00, затем следующая задача.

## Отчёт о прогрессе

После КАЖДОЙ решённой задачи — вывести пользователю краткий отчёт в формате:

```
### tXX — [Score]
- Попыток: N
- Проблемы: (если были — кратко что пошло не так)
- Исправления: (если были — что именно поправлено в системе)
- Время: ~Ns
```

В конце серии — сводная таблица всех задач.

## Запрещено

- Добавлять в системные промпты агентов конкретные имена файлов, пути, значения из задач
- Подгонять логику под частный случай одной задачи
- Менять `.env` без явного запроса пользователя
- Запускать задачи параллельно (только последовательно)
- Пропускать задачу без попытки решения
- Игнорировать score_detail — это главный источник диагностики

## Разрешено и рекомендовано

- Читать и анализировать логи в `logs/`
- Расширять универсальные правила в промптах агентов
- Исправлять Pydantic-модели если валидация блокирует работу
- Увеличивать лимиты (retries, max steps) если агент не успевает
- Добавлять универсальные протоколы в execution_agent.py

## Архитектура системы (краткая справка)

- `main.py` — точка входа, запуск задач через BitGN API
- `orchestrator.py` — стейт-машина: triage → bootstrap → task_model → execution loop
- `llm_provider.py` — провайдеры LLM (claude, openrouter, antigravity, opencode)
- `agents/execution_agent.py` — основной промпт планировщика и вызов LLM
- `agents/types.py` — Pydantic-модели (NextStep, TaskModel, TriageDecision, etc.)
- `agents/triage_node.py` — классификация безопасности
- `agents/task_node.py` — извлечение структуры задачи
- `agents/validator_node.py` — валидация мутаций
- `agents/tool_executor.py` — исполнение инструментов через PCM
