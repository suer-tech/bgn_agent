## Ключевое наблюдение

**System prompt = 73% контекста, history = 27%**. System prompt **одинаковый на каждой итерации** — он пересобирается заново, но содержимое не меняется (кроме scratchpad). На 18-шаговую задачу LLM получает **~6,600 × 18 = ~119,000 prompt-токенов** только на повторяющийся промпт.

## Варианты оптимизации

### 1. Убрать дублирование system prompt (экономия: ~60-70% токенов)

**Сейчас**: `build_planner_prompt()` создаёт полный system prompt каждую итерацию, и вся conversation_history добавляется после.

**Оптимизация**: Разделить на **статический system prompt** (отправляется один раз) и **динамический user message** (scratchpad + entity context). Для API-провайдеров (OpenRouter) это нативно — system message кешируется. Для `claude -p` нет такой возможности, но OpenRouter с `gpt-4.1` поддерживает prompt caching.

### 2. Сжатие workspace rules (экономия: ~2,000-3,000 токенов)

**Сейчас**: Все 9 файлов правил вставляются целиком (14K chars).

**Оптимизация**:
- На этапе bootstrap LLM уже видел эти правила. Вместо полного текста — **дайджест правил**: ключевые правила в 2-3 строки из каждого файла.
- Или: оставить полный текст только ROOT AGENTS.md, а PROCESS/FOLDER — только заголовки + релевантные строки (отобранные по task_model.domain).

### 3. Сжатие conversation_history (экономия: ~30-50% от history)

**Сейчас**: Каждый tool result хранится целиком (до 10K символов каждый).

**Оптимизация**:
- **Sliding window**: хранить только последние N шагов (5-7), старые — суммаризировать в 1-2 строки.
- **Сжатие tool results**: read-результаты после обработки заменять на краткую сводку ("read /contacts/mgr_002.json → found email: x@y.com, account_id: acct_007").
- **Удалять tree/list результаты** из старых шагов — они уже отработаны.

### 4. Entity Context на лету вместо upfront (экономия: ~1,200 токенов)

**Сейчас**: `context_gatherer` собирает таблицы дат заранее и вставляет в каждый промпт.

**Оптимизация**: Не вставлять в промпт — пусть LLM сам вызывает search/read когда нужно. Таблицы дат занимают ~1,200 токенов и повторяются 18 раз.

### 5. Meta-rules: вынести в system prompt один раз (экономия: ~1,000 токенов)

**Сейчас**: 14.8% промпта — это meta-rules, которые не меняются.

**Оптимизация**: Если использовать OpenRouter API — meta-rules в system message (кешируется). Для других провайдеров — сократить текст: многие правила уже понятны модели (MINIMAL DIFF, ENTITY VERIFICATION).

### 6. Prompt caching (экономия: ~50% стоимости, не размера)

OpenRouter/OpenAI поддерживают automatic prompt caching для длинных промптов. Если system prompt >1024 токенов и повторяется — кешируется автоматически. Это не уменьшает context window, но сильно снижает стоимость и latency.

## Рекомендованный план оптимизации (по приоритету)

| # | Действие | Экономия | Сложность | Риск |
|---|---|---|---|---|
| 1 | Sliding window на history (5 шагов + summary) | ~30% history | Средняя | Низкий |
| 2 | Сжатие tool results в history | ~40% history | Низкая | Низкий |
| 3 | LLM-суммаризация rules в bootstrap | ~40% от rules | Средняя | Средний (потеря нюансов) |
| 4 | Убрать entity_context из промпта | ~1,200 токенов × N | Низкая | Средний (LLM потратит шаги на search) |
| 5 | Разделить static/dynamic prompt | ~60% prompt tokens (кеш) | Средняя | Низкий |
